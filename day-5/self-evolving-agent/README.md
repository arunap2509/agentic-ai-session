# Self-Evolving Agent

Every other agent in this repo (day-3/coding-agent, day-4's incident-commander,
etc.) ships with a fixed toolset chosen up front. This one ships with almost
nothing - three small meta-tools - and writes its own tools at runtime when a
task needs a capability it doesn't have yet.

Ask it to do something that needs shell access and it doesn't fail, and it
doesn't ask you to go add a tool: it calls its `create_and_register_new_tool`
meta-tool with a hand-written `run_bash_command` implementation, gets it
loaded and active in the same process, and calls it on its very next turn.
Every tool it writes is saved to `tools/` and reloaded automatically the next
time you run the script, so the toolset only ever grows across sessions.

## The three seed tools

- **`create_and_register_new_tool(name, code, description)`** - the
  self-evolving part. Validates `name` is a legal identifier, writes `code`
  to `tools/<name>.py`, imports it with `importlib`, pulls out the function,
  and adds it to `self.active_tools`. If the generated code is broken
  (syntax error, wrong function name, non-callable), it reports the exact
  error back to the model instead of crashing, so the model can patch it and
  retry - the system prompt tells it to do exactly that.
- **`execute_one_time_script(code, description)`** - for problems that need
  code execution but don't deserve a permanent tool (a calculation, a quick
  data transform). Runs the script, captures stdout/stderr, returns the
  output, registers nothing.
- **`fetch_webpage_content(url)`** - one ordinary, pre-built tool, included
  so there's a visible contrast in the output between "tool I shipped with"
  and "tool the model wrote for itself five seconds ago."

## Why this needed its own loop instead of day-4's `common/agent_loop.py`

`run_tool_loop` in day-4 builds its `types.Tool` function-declaration list
**once**, before the loop starts. That's fine when the toolset is fixed for
the whole run, but it means a tool registered mid-run would never be visible
to the model - the whole point here. `agent.py`'s loop rebuilds the tool
declarations from `self.active_tools` on **every** step
(`SelfEvolvingAgent._build_tool()`), so a tool created on step 1 is callable
on step 2 of the same task.

## Run it

```
cd day-5
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # paste in your GEMINI_API_KEY

cd self-evolving-agent
python main.py
```

Try, in order (each builds on the last):

1. `List the files in this directory using a real shell command.`
   It has no shell tool yet, so it writes `run_bash_command` via the
   meta-tool, then calls it - watch the "Meta-tool Invoked" banner, the
   saved-file path, the inspected signature, then the very next step calling
   the tool it just made.
2. `Now use that same tool to print the first 5 lines of agent.py.`
   Reuses the tool from step 1, no re-creation.
3. `What's 47 * sqrt(193), rounded to 2 decimal places?`
   Doesn't deserve a permanent tool - watch it reach for
   `execute_one_time_script` instead of `create_and_register_new_tool`.
4. Quit (blank input) and run `python main.py` again. It restores
   `run_bash_command` from `tools/run_bash_command.py` before you type
   anything - the "self-evolving" part persists across process restarts,
   not just within one session.

Every run ends with a session summary: how many turns of conversation were
kept, which tools were created this session, and the full current toolset.

## Safety

`create_and_register_new_tool` and `execute_one_time_script` both execute
model-written Python with no sandboxing beyond a subprocess timeout - the
same posture as day-3/coding-agent's `run_bash`. Fine for a local demo where
you control what you ask it to do. Do not point this at untrusted input, and
do not run it anywhere with access to something you'd mind losing - there is
no container/VM boundary here, only a timeout.

## Files

- `agent.py` - `SelfEvolvingAgent` class: the three seed tools, tool
  persistence/restore, and the step loop.
- `main.py` - REPL entry point.
- `tools/` - where dynamically created tools land (gitignored except
  `__init__.py` - they're regenerated by using the agent, not checked in).
